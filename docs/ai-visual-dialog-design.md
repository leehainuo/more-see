# AI 视觉对话助手设计文档

## 1. 文档目的

本文档用于作为“题目一：AI 视觉对话助手”的正式交付材料，重点说明：

- 计划实现哪些用户故事，最终实现了哪些
- 想到了哪些控制运营成本的技巧，实际采用了哪些

## 2. 项目目标

开发一款可以在浏览器中运行的 AI 视觉对话应用。用户打开摄像头与麦克风后，AI 能同时理解用户语音与当前画面内容，并给出自然、及时、可靠的回应。

本项目采用 Web 端前端采集 + FastAPI WebSocket 编排 + 火山引擎 ASR / Vision / LLM / TTS 的端云协同架构，重点突出：

- 多模态主链路完整
- 流式反馈自然
- 成本控制可解释
- 演示稳定性较强

## 3. 用户故事

相比通用的“语音识别 + 图片问答”式描述，本项目的用户故事设计更强调三类真实使用者：

- 普通用户：关注是否能自然地与 AI 进行语音和视觉对话
- 演示者 / 知识工作者：关注是否既能看现实世界，也能看数字世界，并适合讲解、教学和演示
- 管理者：关注系统是否能解释成本、验证优化效果，并支持后续运营决策

这样的角色划分更能体现本项目的核心竞争力：它不是一个单次截图问答 demo，而是一个具备实时会话、双视觉输入、自然打断和成本可解释能力的 Web 端多模态助手。

### 3.3 当前版本的额外亮点

从面试作品角度看，本项目最有吸引力的不是“接了几个模型”，而是以下几个更有区分度的亮点：

- 实时多模态会话闭环：不是截图问答，而是浏览器持续采集、持续会话、持续流式反馈
- 双视觉输入能力：既能看摄像头里的现实物体，也能看屏幕共享中的页面与代码
- 自然语音交互：支持边播报边打断继续说，更接近真实语音助手
- 意图感知回答：会根据用户问题自动切换到科普介绍、文字翻译、内容回复等更贴合场景的回答方式，而不是把所有视觉输入都当作通用聊天处理
- 成本可解释：不仅做了成本优化，还把成本面板、命中情况和优化策略显式呈现出来
- 工程稳定性：具备缓存、降级、自检与历史会话，完成度高于一般竞赛或面试 demo

其中，“意图感知回答”是当前版本一个比较有区分度的增强点。

在当前版本里，Agent 增加了一层轻量意图识别，用于把用户问题路由到更合适的回答策略。当前已经落地的专项意图包括：

- 科普介绍：当用户拿着物体发问，且问题明显带有“介绍一下、做什么用、科普一下”倾向时，系统会把回答组织成“先识别，再分点介绍”的风格
- 文字翻译：当用户问题带有“翻译、什么意思、帮我翻成”倾向时，系统会优先要求视觉链路抽取可见原文，再生成忠实翻译
- 内容回复：当用户给出聊天截图、消息界面或评论内容，并询问“我要怎么回复对方”时，系统会优先提炼对方诉求，再生成 2-3 个可直接发送的回复版本

这样做的意义在于，同样是“看图回答”，不同任务的成功标准并不一样：

- 科普类问题重在“识别后解释得更丰富”
- 翻译类问题重在“原文提取是否准确”
- 回复类问题重在“是否提炼了截图语义，并给出可操作输出”

因此，意图器本质上不是锦上添花的小功能，而是把多模态输入转化为更稳定产品体验的一层任务路由。

## 4. 成本控制技巧

对于这道题，成本控制不是附加项，而是决定作品是否“像真实产品”的关键指标之一。

如果系统只是把连续音频、连续视频和完整上下文全部上传到云端，虽然也能完成问答，但会同时带来三个问题：

- 云侧调用次数高，运营成本迅速上升
- 响应链路变长，用户体感延迟增加
- 系统稳定性更依赖网络和模型状态，现场演示更容易翻车

因此，本项目的成本策略不是单纯追求“最省钱”，而是在三个目标之间做平衡：

- 尽量少传：让端侧先过滤无价值的数据
- 尽量少算：让云侧尽量复用已有结果
- 尽量不断：即使某一环节变慢，也不要拖垮整轮对话

### 4.1 想到过的成本控制技巧

| 编号   | 技巧                 | 是否采用 | 主要作用                     | 取舍说明                                |
| ---- | ------------------ | ---- | ------------------------ | ----------------------------------- |
| C-01 | 音频静音裁剪             | 已采用  | 减少 ASR 无效音频上送与计费时长       | 为避免漏字，保留预滚动和尾静音窗口，不走极限压缩方案          |
| C-02 | 关键帧相似复用            | 已采用  | 减少重复关键帧上传和重复视觉调用         | 画面变化极小但语义变化较大的场景，仍可能需要重新捕获          |
| C-03 | 服务端视觉摘要缓存          | 已采用  | 对重复图像内容直接复用摘要结果          | 命中效果依赖缓存大小和图像一致性                    |
| C-04 | ASR 与视觉并行 + 视觉超时降级 | 已采用  | 保证低延迟主链路，不让视觉慢请求阻塞回复     | 个别轮次会先基于语音回答，再等待下一轮获取更强视觉信息         |
| C-05 | 最小上下文窗口            | 已采用  | 控制 LLM 上下文长度，减少 token 消耗 | 长会话场景下需要依赖摘要或记忆模块弥补                 |
| C-06 | 模型路由               | 未采用  | 本可进一步降低复杂问题以外的模型成本       | 当前阶段优先统一链路和演示稳定性，暂不引入额外决策复杂度        |

### 4.2 实际采用的成本控制策略

本项目最终采用的策略可以归纳为三层：端侧减少无效数据、服务端复用已有结果、系统级保证低延迟不断链。

#### 1. 端侧音频静音裁剪

- 方案：前端以音量阈值识别有效语音，仅在有效语音段和尾静音窗口中继续上送音频 chunk
- 实际价值：减少无意义静音片段进入火山 ASR，降低语音识别成本与上行带宽
- 设计取舍：为了尽量不漏字，保留了预滚动与尾静音窗口，因此这是一种“稳字优先”的压缩策略，而不是单纯追求最低成本

#### 2. 端侧关键帧相似复用

- 方案：前端对关键帧计算差分指纹（dHash），相似画面直接复用上一轮视觉摘要
- 实际价值：减少重复上传与重复视觉调用，尤其适合“同一物体连续追问”或“画面基本静止”的场景
- 设计取舍：画面变化很小但语义变化大的极端场景，仍可能需要用户重新调整镜头，因此这是成本与准确率之间的平衡策略

#### 3. 服务端视觉摘要缓存

- 方案：后端对关键帧内容做哈希缓存，重复内容直接复用摘要
- 实际价值：降低重复视觉推理成本，提升返回速度，让“重复看同一个画面”的成本更可控
- 设计取舍：缓存命中依赖图像一致性与缓存大小策略，因此它不是端侧门控的替代，而是第二层保险

#### 4. 视觉并行与超时降级

- 方案：ASR 与视觉摘要并行；若视觉未就绪，则优先继续语音问答
- 实际价值：保证低延迟体验，避免某一环节拖慢整轮，让系统更适合真实演示和实际使用
- 设计取舍：个别轮次可能暂时只基于语音回答，但整体体验会明显优于“整轮卡住等待视觉”

#### 5. 最小上下文窗口

- 方案：仅保留最近几轮必要会话历史，避免每轮都把所有历史送给 LLM
- 实际价值：减少 token 消耗与推理延迟，避免历史不断堆积导致每一轮都越来越贵
- 设计取舍：超长会话需要依赖摘要或记忆模块补充，因此它更适合当前“演示与高频问答”场景

## 5. 端云协同设计总结

### 5.1 端侧负责

- 设备权限申请与音视频采集
- 音频下采样、静音检测、音频 chunk 上送
- 关键帧抓取、相似帧判断与上传
- 工作台状态反馈、流式文本展示与音频播放

### 5.2 云侧负责

- WebSocket 会话编排
- 火山 ASR / Vision / LLM / TTS 调用
- 意图识别与多模态任务路由
- 多模态上下文拼装、历史管理与降级处理
- 成本统计、缓存与可观测性沉淀

### 5.3 为什么当前不把 YOLO 作为主方案

- 当前题目强调“看懂 + 听懂 + 自然回应”，主角应是多模态理解链路，而不是目标检测本身
- 对 Web MVP 而言，引入 YOLO 会增加端侧负载、包体与调试复杂度
- 当前采用“关键帧 + 摘要 + 相似复用 + 缓存”的方案，已经更适合本项目的成本与延迟目标
